# AGENTS.md

本文件是给 Codex / agent 使用的项目级行为规则，不是普通项目介绍。项目介绍请阅读 `PROJECT_CONTEXT.md` 和 `README.md`。

## 每次开始工作前

1. 必须先阅读 `PROJECT_CONTEXT.md`。
2. 必须先阅读 `wiki/current-status.md`。
3. 如任务涉及架构、类别、数据、硬件或接口，还需要阅读对应 wiki 文档。
4. 工作必须 step by step：先说明计划，再进行实现或文档修改。

## 工作语言

- 本项目默认使用中文沟通、记录项目文档和更新 Obsidian/wiki。
- 文件名、文件夹名、代码变量、配置字段、API 字段、模型名可以使用英文。
- 除非用户明确要求，Markdown 文档内容以中文为主。

## 范围控制

- 不要擅自实现用户没有要求的功能。
- 不要把初步技术方向当成已经确认的工程事实。
- 不要假设机器人接口、Thor 环境、Orange Pi 环境、训练数据或类别清单已经确定。
- 不要擅自更改硬件部署方案、机器人通信方案或项目职责边界。
- 如果任务描述不明确，先提出选项或记录到 `wiki/open-questions.md`，不要自己编答案。

## 禁止擅自执行的操作

除非当前用户请求明确要求或批准，不要执行以下操作：

- 安装依赖。
- 创建 Python 虚拟环境。
- 训练模型。
- 下载大模型、数据集或模型权重。
- 启动本地服务。
- 编写或运行硬件部署流程。
- 自动 commit、push 或 deploy。

这些是长期权限边界，不是永久禁止功能开发。后续如果用户明确要求进入 YOLO baseline、训练脚本、推理脚本、API demo 或部署验证阶段，可以在确认范围和风险后按步骤执行。

## 数据与模型文件

- 不要 commit 大型数据集、模型权重、图片、视频、标注导出包或实验输出。
- 数据目录、模型权重、导出文件和实验结果应被 `.gitignore` 忽略。
- 如需要保留目录结构，使用轻量说明文件，例如 `.gitkeep` 或 `README.md`。

## 文档维护规则

- 每次 meaningful change 后必须更新 `wiki/current-status.md` 和 `wiki/codex-log.md`。
- 架构选择必须记录到 `wiki/decisions.md`。
- 不确定的问题必须记录到 `wiki/open-questions.md`，不要自行假设答案。
- 类别、数据、硬件和接口相关变更应同步更新对应 wiki 文档。

## 临时限制的处理方式

- 如果当前用户请求包含本轮 pass 的临时限制，必须严格遵守。
- 不要把某一次 setup pass、调研 pass 或修正 pass 的临时限制写成项目永久规则。
- 临时限制只约束当前请求；长期规则以本文件的权限边界、范围控制和文档维护规则为准。
